# 2026-03-03 ORT Runtime Collision Analysis

## 요약

- 최신 덤프에서 `ZSoda.aex` 내부 `C0000005 (read @0x18)` 크래시가 반복됨.
- 관측된 프로세스 모듈에는 Adobe 경로의 `onnxruntime.dll (1.17.24.223)`가 존재하며,
  플러그인 빌드 기준 ORT(1.24.2) DLL은 로드 흔적이 없음.
- 결론: **ORT 런타임 버전/로딩 경로 충돌 가능성이 매우 높음**.

## 근거

1. Faulting module이 `ZSoda.aex`이며 read access violation 주소가 `0x18` 오프셋 패턴.
2. ORT C++ API 초기화/호출 지점에서 API 테이블 협상 실패 시 null 역참조 패턴과 부합.
3. AE 프로세스 특성상 타 Adobe 모듈이 먼저 ORT를 로드해 심볼 해석이 오염될 수 있음.

## 결정 사항

- 단기 핫픽스보다 **구조적 해결**을 우선한다.
- 목표는 "어떤 환경에서도 ORT 로드 소스/버전이 결정적으로 통제되는 것"이다.

## 구조적 해결 원칙

1. 암시적 ORT 링크(onnxruntime.lib 의존)에만 기대지 않는다.
2. 런타임에서 ORT를 **명시적 로딩**하고, 실제 로드된 DLL 경로를 기록한다.
3. `OrtGetApiBase()->GetApi(ORT_API_VERSION)` 협상 실패 시 즉시 추론 비활성화 + 안전 폴백.
4. 예외/크래시 대신 진단 가능한 오류 문자열을 `fallback_reason`에 남긴다.

## 실행 작업(멀티 에이전트 위임)

- 빌드/패키징: 구조적 로딩 옵션 추가, ORT DLL 배치 강제 검증.
- 추론 런타임: 동적 ORT 로더 유틸 추가(경로/버전/실패원인 진단 포함).
- 백엔드 통합: OnnxRuntimeBackend 초기화 경로 리팩터링.
- 테스트: ORT 불일치 시 크래시 대신 폴백/진단 문구 검증.
- 운영 문서: 로컬 핸드오프 및 빌드 가이드에 절차 반영.

## 완료 기준

1. ORT 버전 충돌 환경에서도 AE 크래시가 재현되지 않는다.
2. `fallback_reason`/로그로 로드 경로 및 실패 사유가 확인된다.
3. AE 스모크 테스트(ST-01~ST-07)에서 최소 ST-01/03/04/06 통과.
